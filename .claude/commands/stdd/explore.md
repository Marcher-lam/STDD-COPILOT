# /stdd:explore - 自由探索模式

> 进入只读探索模式，在动手前理清思路。探索**业务代码**和架构，可衔接 `/stdd:new`。

## ⚠️ 重要提示

**探索目标** 是必需的。如果没有提供，请立即询问用户：

```
🔍 请告诉我您想探索什么？

示例：
- "理解现有的用户认证流程"
- "分析 API 错误处理模式"
- "评估引入状态管理库的可行性"
- "理解支付模块的架构设计"
```

## 使用方式

```
/stdd:explore <探索目标>
```

**必须**提供探索目标。探索目标描述你想要理解的**业务功能**或**技术问题**。

## 执行步骤

### Step 1: 确认探索目标

如果用户没有提供探索目标，立即询问。不要自己猜测。

### Step 2: 分析业务代码

**只探索业务代码**，不要探索 STDD 配置文件。

```
✅ 探索这些:
   - src/ 目录下的业务代码
   - 现有的组件、服务、工具
   - API 接口和数据模型
   - 测试文件和测试模式

❌ 不要探索这些:
   - .claude/ (STDD 命令配置)
   - stdd/ (STDD 工作目录)
   - schemas/ (规格模板)
   - AGENTS.md (AI 指令)
```

### Step 3: 识别约束

分析项目的技术约束：
- 框架和库 (React, Vue, Express 等)
- 语言和版本 (TypeScript, Node.js 版本)
- 编码规范和风格
- 现有架构模式

### Step 4: 探索方案

如果探索目标涉及技术选型或实现方案：
- 列出 2-3 种可能的实现方式
- 分析每种方案的优缺点
- 给出推荐方案和理由

### Step 5: 生成报告

创建探索报告到 `stdd/explorations/`:

```markdown
# 探索报告: [探索目标]

## 探索时间
[时间戳]

## 发现

### 当前架构
[描述发现的架构模式]

### 关键文件
- `path/to/file1.ts` - [说明]
- `path/to/file2.ts` - [说明]

### 技术约束
- [约束 1]
- [约束 2]

## 分析

### 方案对比 (如适用)
| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案A | ... | ... |
| 方案B | ... | ... |

### 推荐方案
[推荐方案和理由]

## 建议

### 下一步
1. [建议 1]
2. [建议 2]

### 可衔接到
`/stdd:new [基于探索的提案描述]`
```

## 输出文件

```
stdd/explorations/
└── exploration-YYYYMMDD-HHMMSS.md
```

## 示例

```
# 探索现有架构
/stdd:explore 理解现有的用户认证流程

# 探索数据访问模式
/stdd:explore 分析数据库查询模式和 ORM 使用情况

# 技术选型探索
/stdd:explore 评估引入 Zustand 替代 Redux 的可行性

# 理解特定模块
/stdd:explore 理解支付模块如何与第三方支付网关集成
```

## 适用场景

- **需求模糊** - 还没想清楚要做什么
- **不熟悉代码** - 接手新项目或模块
- **技术选型** - 需要评估多个方案
- **重构准备** - 动手前先理解现状
- **Bug 分析** - 理解问题根源

## 禁止行为

- ❌ 修改任何代码
- ❌ 创建或删除文件
- ❌ 探索 STDD 配置文件
- ❌ 在没有目标时自行猜测
